# 紀錄

## 單一 worker

當訂單集中購買相同商品時，樂觀鎖衝突率大幅上升，每次衝突都需等待後重試，導致 worker 整體吞吐量下降。

在此條件下，增加 worker 數量不一定能提升效能，反而可能因大量並行衝突而使 DB 壓力上升、整體處理時間拉長。因此在「單一熱點商品搶購」情境中，使用單一高規格 worker 搭配嚴格的重試策略，反而可能比多 worker 更穩定。

> **前提**：此結論成立的關鍵條件為衝突率極高（所有訂單爭同一商品），若商品有多個庫存分片或訂單分散，結論會不同。


## 多 worker

在一般電商日常中，訂單同時碰撞到相同商品的機率極低（假設品項多且購買分布均勻），此時增加 worker 數量可有效提升訂單處理吞吐量。

### 測試參數

| 項目 | 值 |
|------|----|
| 商品種類 | 50 種 |
| 每種商品庫存 | 200 個 |
| 訂單總數 | 5,000 張 |
| 每張訂單商品數 | 2 種不同商品各 1 個 |
| 總消耗庫存 | 10,000 件（剛好賣完） |
| worker 規格 | 所有 worker 效能一致 |

### 測試結果

| worker 數 | 處理時間 |
|-----------|---------|
| 1 | 約 60 秒 |
| 5 | 約 12 秒 |

吞吐量提升約 5 倍，接近線性擴展。

> **注意**：線性擴展僅在此低衝突測試條件下成立。在實際環境中，隨 worker 數繼續增加，DB 連線池、Redis、RabbitMQ 等共享資源將成為瓶頸，擴展效益會逐漸遞減（邊際報酬遞減）。
>
> 本測試僅執行單次，若要作為決策依據，建議多次執行取平均值並觀察標準差，同時記錄失敗訂單數與樂觀鎖重試次數。
